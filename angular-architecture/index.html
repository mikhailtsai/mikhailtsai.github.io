File structure The idea is to write isolated modules with controlled dependency reuse
apps
example-app
src
app.component.ts — has no logic, is used only for some required integrations in the root component
libs
navigation — if the application has a common navigation logic that is used across all pagescommon — directory
ui — storing small and reusable componentstools — storing common services, helpers, consts, models
accounts — feature directory example split by 3 libraries: view, data and model
profile-view — presentation layer of the featureprofile-data — data layer of the featureprofile-model — if profile-data is sharable between another modules we add this library to store DI token and another models
Feature directory structure Should I always split a feature into 3 separate libraries? No.

1 feature — 1 library:

If the feature itself is small, does not require state storage, has only presentation logic and reuse of other libraries.

libs
some-feature — feature library

You can use this type of feature as a route:

app.routes.ts

import { Route } from '@angular/router';

export const appRoutes: Route[] = [
  {
    path: '',
    loadChildren: () =>
      import('@example-app/some-feature').then((module) => module.someFeatureRoutes),
  },
];

You can also use this type of feature as a component in another modules:

some-another-feature.component.ts

import { SomeFeatureWidgetComponent } from '@example-app/some-feature';

@Component({
  selector: 'app-some-another-feature',
  standalone: true,
  imports: [
    SomeFeatureComponent,
  ],
  template: `<app-some-feature-widget />`,
  changeDetection: ChangeDetectionStrategy.OnPush,
})
export class SomeAnotherFeatureComponent {}
